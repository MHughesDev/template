**Purpose:** [FULL SKILL] How to roll out database migrations safely using the expand/contract pattern, with CI dry-run, production approval gates, and rollback plan. Per spec §26.4 item 70.

## Purpose

One paragraph. Database migrations in production carry significant risk — wrong migrations can corrupt data, lock tables, or require downtime. The expand/contract pattern separates risky operations (DROP, NOT NULL, RENAME) into two phases, each safe to deploy independently. This skill makes migration risk explicit and manageable.

## When to Invoke

- When any Alembic migration is being authored
- When a migration requires a NOT NULL constraint, column drop, or column rename
- When a migration will touch a table with millions of rows (index creation strategy)
- Before any production migration deployment

## Prerequisites

docs/procedures/database-migration.md read. prompts/migration_author.md available for reference. Local database running and accessible.

## Step-by-Step Method

Numbered steps:

**Phase analysis (before writing code)**:
1. Classify the change:
   - EXPAND (safe alone): ADD column (nullable or with default), ADD table, ADD index, ADD enum value
   - CONTRACT (risky, needs expand first): DROP column, DROP table, ADD NOT NULL without default, RENAME
   - Two-phase: risky changes require expand first, then contract in a later deployment

**Writing the migration**:
2. `make migrate:create MESSAGE="expand: add status column to invoices"` or `"contract: drop legacy column"`
3. Review autogenerated file — Alembic is not always correct; fix manually as needed
4. Write explicit upgrade(): specific operations in safe order
5. Write explicit downgrade(): exact reverse of upgrade (test this!)
6. For large tables: use `op.execute("CREATE INDEX CONCURRENTLY ...")` (PostgreSQL)
7. For NOT NULL: expand phase adds nullable column → data migration → contract phase adds NOT NULL constraint

**Testing locally**:
8. `make migrate` — apply upgrade, verify schema
9. `alembic downgrade -1` — verify downgrade works
10. `alembic upgrade head` — reapply, verify idempotent behavior

**CI dry-run**:
11. `alembic upgrade head --sql > migration.sql` — review generated SQL
12. Verify SQL in PR description for reviewer

**Production deployment**:
13. Expand phase: deploy without application code change (safe)
14. Deploy application code that works with both old and new schema
15. Contract phase: deploy in a later release after verifying expand phase is stable

## Command Examples

- `make migrate:create MESSAGE="expand: add status column"` — scaffold revision
- `make migrate` — apply migration
- `alembic downgrade -1` — rollback one step
- `alembic history` — show migration history
- `alembic upgrade head --sql` — dry-run (prints SQL only)

## Validation Checklist

- [ ] Change classified as expand, contract, or two-phase
- [ ] upgrade() verified (make migrate succeeds)
- [ ] downgrade() verified (alembic downgrade -1 succeeds)
- [ ] SQL dry-run reviewed in PR description
- [ ] Data migration included if NOT NULL constraint added
- [ ] Large table strategy (CONCURRENTLY) used if applicable
- [ ] Rollback plan documented in PR

## Common Failure Modes

- **Missing downgrade**: `alembic downgrade -1` fails → migration is non-reversible → blocked from production. Fix: always write and test downgrade() before PR.
- **NOT NULL without default**: migration fails on non-empty table. Fix: expand phase adds nullable column, data migration fills it, contract phase adds NOT NULL.
- **Table lock in production**: large table migration causes service degradation. Fix: use `CREATE INDEX CONCURRENTLY` and test on production-sized data.

## Handoff Expectations

Migration file committed, upgrade + downgrade tested locally, SQL dry-run in PR description, rollback plan documented.
