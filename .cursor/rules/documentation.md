> PURPOSE: Rules for when documentation MUST be updated alongside code changes. Prevents documentation drift. Recommended per spec §26.2 item 16.

## Section: Environment Variable Trigger

> CONTENT: Rules triggered by env var changes. Rules:
> 1. Adding a new env var: MUST update `.env.example` (with comment and placeholder) AND `docs/development/environment-vars.md`
> 2. Removing an env var: MUST remove from `.env.example` AND `docs/development/environment-vars.md`
> 3. Changing an env var's meaning or format: MUST update both `.env.example` comment AND the docs entry
> 4. Run `scripts/env-var-sync.py` to verify sync between code and `.env.example`

## Section: API Endpoint Trigger

> CONTENT: Rules triggered by API endpoint changes. Rules:
> 1. Adding a new endpoint: MUST update `docs/api/endpoints.md` with method, path, description, auth, schemas
> 2. Removing an endpoint: MUST update `docs/api/endpoints.md` and add deprecation notice if previously public
> 3. Changing endpoint behavior (request/response schema, auth requirement): MUST update `docs/api/endpoints.md`
> 4. New error code: MUST add to `docs/api/error-codes.md` with HTTP status, description, client guidance
> 5. OpenAPI baseline: run `scripts/openapi-diff.py` before releases to detect breaking changes

## Section: Behavior Change Trigger

> CONTENT: Rules triggered by changes to system behavior. Rules:
> 1. Configuration change: MUST update `docs/operations/configuration.md`
> 2. Docker/Compose change: MUST update `docs/operations/docker.md`
> 3. Kubernetes manifest change: MUST update `docs/operations/kubernetes.md`
> 4. Auth or security behavior change: MUST update relevant `docs/security/` file
> 5. Queue lifecycle change: MUST update `queue/QUEUE_INSTRUCTIONS.md` AND `docs/queue/queue-system-overview.md`
> 6. Architectural decision: MUST create ADR in `docs/adr/` using the template

## Section: Procedure Drift Trigger

> CONTENT: Rules for keeping procedures current. Rules:
> 1. When a procedure's Make target changes: update the procedure file with the new command
> 2. When a procedure's steps change order: update the procedure file
> 3. When a prerequisite for a procedure changes: update the Prerequisites section
> 4. After changing any procedure: run `make docs:check` to verify no broken links
> 5. Quarterly: run `scripts/docs-freshness-checker.py` to surface stale documentation

## Section: Documentation Generation Pipeline

> CONTENT: Rules for the auto-generation pipeline. Rules:
> 1. Source-linked documentation (endpoint catalog, env var reference, error codes) is GENERATED by `make docs:generate`
> 2. Never manually edit generated sections — edit the source and regenerate
> 3. CI runs `make docs:check` which verifies generated docs match the current source
> 4. Drift between generated docs and source is a CI failure
> 5. When adding new auto-generated doc targets: extend `skills/repo-governance/docs-generator.py`
